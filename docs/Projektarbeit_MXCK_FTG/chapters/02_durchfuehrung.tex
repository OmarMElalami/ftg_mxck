\chapter{Systembeschreibung und Durchführung}
\label{ch:durchfuehrung}

Dieses Kapitel beschreibt die tatsächlich umgesetzte FTG-Architektur im
aktuellen Repository \texttt{ftg\_mxck}. Der Fokus liegt dabei auf dem
scan-basierten Standardpfad, der gegenüber älteren Legacy-Varianten mit
\texttt{obstacle\_substitution} bewusst bevorzugt wird.

\section{Zielarchitektur des aktuellen Stacks}

Die aktuelle Root-Dokumentation des Repositories definiert die Primärpipeline
klar als direkten Scan-Pfad bis zur Ackermann-Ausgabe. Daraus ergibt sich
folgende Soll-Architektur:

\begin{lstlisting}
 /scan (LaserScan, frame: laser)
   -> scan_preprocessor_node               [mxck_ftg_perception]
   -> /autonomous/ftg/scan_filtered        [LaserScan, frame: base_link]
   -> /autonomous/ftg/front_clearance      [Float32]

 /autonomous/ftg/scan_filtered
   -> follow_the_gap_v0                    [C++, input_mode=scan]
   -> /final_heading_angle                 [Float32]
   -> /gap_found                           [Bool]

 /final_heading_angle + /gap_found + /autonomous/ftg/front_clearance
   -> ftg_planner_node                     [mxck_ftg_planner]
   -> /autonomous/ftg/gap_angle            [Float32]
   -> /autonomous/ftg/target_speed         [Float32]
   -> /autonomous/ftg/planner_status       [String]

 /autonomous/ftg/gap_angle + /autonomous/ftg/target_speed
   -> ftg_command_node                     [mxck_ftg_control]
   -> /autonomous/ackermann_cmd            [AckermannDriveStamped]
\end{lstlisting}

Wichtig ist dabei, dass die Ackermann-Ausgabe bewusst nicht direkt in
niedrigere VESC-Topics schreibt, sondern sich in die bestehende MXCK-
Fahrzeugsteuerung einfügt.

\section{Stufe 1: LiDAR-Vorverarbeitung}

\subsection{\texttt{scan\_preprocessor\_node}}

Der \texttt{scan\_preprocessor\_node} bildet den Eintrittspunkt des
Produktivpfades. Seine Aufgabe besteht darin, aus dem rohen
\texttt{/scan}-Topic einen für den FTG-Kern gut interpretierbaren
Front-Scan zu erzeugen.

Dazu führt der Node folgende Schritte aus:

\begin{enumerate}
    \item Abonnement von \texttt{/scan},
    \item TF-Lookup von \texttt{base\_link <- laser},
    \item Bestimmung der Fahrtrichtung im Basisrahmen,
    \item Auswahl des Frontfensters (z.\,B.\ 100°),
    \item Clipping auf gültige Minimal- und Maximalreichweiten,
    \item Publikation des gefilterten Scans auf
    \texttt{/autonomous/ftg/scan\_filtered},
    \item Publikation des minimalen Frontabstands auf
    \texttt{/autonomous/ftg/front\_clearance}.
\end{enumerate}

Die zentrale Designentscheidung ist dabei: \texttt{front\_center\_deg = 0.0}
beschreibt ``vorne im \texttt{base\_link}-Frame''. Die reale Montagerotation des
LiDAR wird ausschließlich durch TF aufgelöst. Dadurch muss kein fester
Sensor-Offset mehr in die Dokumentation oder in die Laufzeitlogik
eingebrannt werden.

\subsection{\texttt{scan\_front\_window\_check}}

Der Diagnose-Node wird optional zugeschaltet und dient nicht der normalen
Fahrt, sondern der Kalibrierung und Verifikation. Er hilft dabei,
das konfigurierte Frontfenster mit der tatsächlichen Fahrzeugfront
abzugleichen und die Wirkung des TF-Setups sichtbar zu machen.

\section{Stufe 2: FTG-Kern im Scan-Modus}

Die aktuelle FTG-Nutzung im MXCK geht bewusst \emph{nicht} über den alten
Obstacle-Substitution-Pfad, sondern direkt über einen vorbereiteten
\texttt{LaserScan}. Der C++-Node \texttt{follow\_the\_gap} wird daher mit
\texttt{input\_mode=scan} gestartet.

\subsection{Ein- und Ausgänge}

Der aktuelle CTU-Kern nutzt konzeptionell:
\begin{itemize}
    \item einen Scan-Eingang,
    \item optional einen Zielwinkel,
    \item einen finalen Heading-Winkel als Ausgabe,
    \item sowie Visualisierungstopics zur Diagnose.
\end{itemize}

Im aktuellen MXCK-Stack werden insbesondere folgende Topics relevant:
\begin{itemize}
    \item Input: \texttt{/autonomous/ftg/scan\_filtered}
    \item Output: \texttt{/final\_heading\_angle}
    \item Output: \texttt{/gap\_found}
    \item Debug: \texttt{/visualize\_obstacles}
    \item Debug: \texttt{/visualize\_largest\_gap}
    \item Debug: \texttt{/visualize\_final\_heading\_angle}
\end{itemize}

\subsection{Bedeutung für das Projekt}

Diese Stufe ist fachlich der Kern des Systems: Hier wird aus der
lokalen Hindernissituation ein Richtungsvektor berechnet.
Die Entscheidung, den Node nicht umzuschreiben, sondern als
C++-Kern beizubehalten und nur sauber scan-basiert einzubinden,
erhält die algorithmische Nähe zur CTU-/F1TENTH-Vorlage.

\section{Stufe 3: Planner}

Der \texttt{ftg\_planner\_node} fasst die algorithmische Ausgabe des FTG-Kerns
mit der Frontabstands-Information aus der Vorverarbeitung zusammen.
Er bildet damit die Brücke zwischen lokaler Richtungsplanung und einer
fahrbaren, sicheren Sollvorgabe.

\subsection{Eingänge}

\begin{itemize}
    \item \texttt{/final\_heading\_angle}
    \item \texttt{/gap\_found}
    \item \texttt{/autonomous/ftg/front\_clearance}
\end{itemize}

\subsection{Ausgänge}

\begin{itemize}
    \item \texttt{/autonomous/ftg/gap\_angle}
    \item \texttt{/autonomous/ftg/target\_speed}
    \item \texttt{/autonomous/ftg/planner\_status}
\end{itemize}

\subsection{Aufgabe des Planers}

Der Planer übernimmt drei praktische Aufgaben:
\begin{enumerate}
    \item Freshness-Prüfung der Eingänge,
    \item Begrenzung des Lenkwinkels,
    \item Geschwindigkeitsvorgabe abhängig von Clearance und Steering.
\end{enumerate}

Damit ist der Planner keine zweite FTG-Implementierung, sondern eine
\emph{Policy-Schicht}: Er interpretiert die Heading-Ausgabe im Kontext von
Sicherheitsabstand und Fahrbarkeit.

\section{Stufe 4: Control}

Der \texttt{ftg\_command\_node} transformiert die Planner-Ausgänge in einen
Ackermann-Befehl.

\subsection{Eingänge und Ausgang}

\begin{itemize}
    \item Input: \texttt{/autonomous/ftg/gap\_angle}
    \item Input: \texttt{/autonomous/ftg/target\_speed}
    \item Output: \texttt{/autonomous/ackermann\_cmd}
\end{itemize}

Zusätzlich veröffentlicht der Node ein Status-Topic
\texttt{/autonomous/ftg/control\_status}, das für Debugging und End-to-End-
Analyse nützlich ist.

\section{Bringup und Startlogik}

Die empfohlene Startdatei ist \texttt{ftg\_full\_system.launch.py}. Laut
aktuellem Repository startet sie den scan-basierten Pfad und lädt dabei
die YAML-Konfigurationen der Perception-, Planner- und Control-Stufe.
Zusätzlich existiert eine vereinfachte Variante
\texttt{ftg\_scan\_path.launch.py}.

Wichtig ist: TF und \texttt{vehicle\_control} gehören \emph{nicht} automatisch
zum FTG-Bringup, sondern werden separat über das MXCK-System gestartet.

\section{Testmethodik}

Für ein robustes Vorgehen wurde der Stack stufenweise verifiziert:

\begin{enumerate}
    \item TF und LiDAR separat prüfen,
    \item nur \texttt{scan\_preprocessor\_node} starten,
    \item \texttt{follow\_the\_gap\_v0} isoliert testen,
    \item Planner prüfen,
    \item Control prüfen,
    \item Full-Stack testen,
    \item erst danach Fahrzeug im Autonomous-Modus fahren.
\end{enumerate}

Diese Reihenfolge ist nicht nur praktisch, sondern auch
architekturell sinnvoll: Jeder Fehler kann dadurch einer klaren
Stufe zugeordnet werden.

\section{Abgrenzung zu Legacy-Komponenten}

Das Repository enthält weiterhin \texttt{obstacle\_msgs} und
\texttt{obstacle\_substitution}. Diese Pakete bleiben aus Gründen der
Kompatibilität erhalten, gehören aber nicht mehr zum primären Datenpfad.
Für die Projektdokumentation bedeutet das:
\begin{itemize}
    \item Sie dürfen als Legacy- oder Alternativpfad erwähnt werden,
    \item sie dürfen aber nicht mehr als aktuelle Hauptpipeline dargestellt werden.
\end{itemize}

Gerade diese Trennung war für die Überarbeitung der Dokumentation zentral, weil
ältere Kapitelentwürfe den Legacy-Pfad teils noch als Hauptarchitektur
beschrieben.
