## 1) So sieht dein **echtes** MXCarkit-System gerade aus

Dein Jetson ist ein **Xavier NX** mit **JetPack 5.1.5 / Ubuntu 20.04**. Auf dem Host laufen mehrere Docker-Container. Für das eigentliche Fahrzeug-Backend ist der Workspace **`~/mxck2_ws`** gedacht; die README beschreibt ihn als Core-Workspace mit `vehicle_control` und `mxck_run`. `mxck_run` ist der zentrale Sensor-/Aktuator-Launcher, und `vehicle_control` ist die Low-Level-Kette Richtung VESC/RC.  

Der LiDAR-Container `mxck2_lidar` ist bereits korrekt an dieses Core-System angebunden: Er bindet **`/home/mxck/mxck2_ws` nach `/mxck2_ws`**, läuft mit **`ROS_DISTRO=humble`** und startet direkt mit
`ros2 launch mxck_run mxck_run_launch.py run_lidar:=true`. Dasselbe steht auch in deiner Compose-Datei für den Service `lidar`. Zusätzlich ist in `.bash_aliases` `run_lidar` genau auf diesen Launch gemappt.   

Der **größte Verwirrungspunkt** ist dein separater Dev-Container `mxck2_development`: Er bindet **nicht** `~/mxck2_ws`, sondern **`~/development_ws`** nach `/mxck2_ws`, und er läuft mit **`ROS_DISTRO=foxy`** in einem Ultralytics/Jetson-Setup. Gleichzeitig laufen `mxck2_lidar` und `mxck2_control` auf dem Core-Workspace mit Humble. Das heißt: Du hattest zuletzt effektiv **zwei verschiedene Workspaces und zwei verschiedene ROS-Distributionen** parallel im Spiel. Das erklärt sehr gut, warum viel Code “gebaut” wurde, aber nicht im realen Fahr-Stack ankam.    

Dazu kommt: Dein `mxck2_control`-Container startet laut Compose/Inspect nur mit `bash`, also **nicht automatisch** mit laufenden ROS-Nodes. Die Steuerkette wird bei dir getrennt gestartet; in deiner Systemdoku war dafür `kickstart` vorgesehen. Dass viele deiner `control_*` Exporte leer waren, liegt schlicht daran, dass dort `ros2` ohne korrekt gesourcte Umgebung aufgerufen wurde und deshalb nur `ros2: command not found` zurückkam. Diese Exporte sind also für die echte Topic-Rekonstruktion **nicht belastbar**.    

## 2) Die wichtigste Entscheidung ab jetzt

Für deine **echte FTG-Integration** solltest du **nur noch einen autoritativen Runtime-Workspace** verwenden:

### **`~/mxck2_ws` ist ab jetzt die Wahrheit.**

Nicht `~/development_ws`.

Der Grund ist einfach: Genau diesen Workspace binden deine realen Runtime-Container `mxck2_lidar`, `mxck2_control`, `mxck2_camera`, `mxck2_micro` und `mxck2_kickstart`. Dein LiDAR läuft dort bereits, und dort hängt auch die bestehende Vehicle-Control-Kette dran.    

`~/development_ws` darfst du weiter als **Lesereferenz** behalten, denn dort liegen schon FTG-nahe Pakete wie `follow_the_gap_v0`, `follow_the_gap_v0_ride`, `obstacle_substitution`, `obstacle_msgs`, `lidar_obstacle_perception`, `mxck_obstacle_substitution` und auch deine fehlgeschlagenen Experimente `ftg_scan_filter` und `lidar_vs`. Aber: **nichts davon** sollte jetzt noch deine eigentliche Implementierungsbasis sein.   

## 3) Deine grobe Idee ist gut — aber die beste Reihenfolge ist etwas besser

Deine Idee war:

1. LiDAR-Steuerung + Scan-Daten
2. Scan → Obstacles
3. FTG
4. Steuerbefehle zur bestehenden Control-Kette

Für dein reales System ist die **bessere** Reihenfolge:

1. **LiDAR nicht neu bauen**, sondern vorhandenen `mxck_run`/`lidar`-Service benutzen
2. **Scan validieren**
3. **Scan vorverarbeiten**
4. **FTG direkt auf gefiltertem Scan**
5. **Steuer-/Safety-Node** → **`/autonomous/ackermann_cmd`**
6. **Obstacle-Layer nur optional für Debug/Visualisierung**
7. **Bringup + Offline-Test + Realtest**

Warum?
Weil das Original-FTG nach Sezer/Gokasan im Kern aus **Gap-Array**, **Maximum-Gap**, **Gap-Center-Angle** und **Final-Heading-Angle** besteht. Dafür brauchst du in der ersten drivable Version **keine eigene Obstacles-Zwischenwelt mit Custom-Messages**. Das Paper betont außerdem die Berücksichtigung von **Field-of-View** und **Nicht-Holonomik** — also genau die Themen, die du zuerst über Scan-Fenster, Sicherheitsabstände und Ackermann-Ausgabe sauber aufsetzen solltest. 

Außerdem muss dein finaler autonome Output in deiner MXCK-Architektur auf **`/autonomous/ackermann_cmd`** landen, damit die vorhandene Steuerkette den Befehl im Autonomous-Modus übernimmt. Das war in deiner Systemdoku bereits der richtige Zielpunkt.

## 5) Der beste Plan – in echten Stages

## Stage 0 — Aufräumen und klare Spielregeln

**Ziel:** Ab jetzt kein weiterer Mischbetrieb mehr.

## Stage 1 — Sensorbasis und Frontfenster

**Ziel:** Erst sicher beweisen, dass `/scan` und die Orientierung stimmen.

Wichtig: Den LiDAR startest du **nicht** mit einem eigenen Node, sondern mit dem vorhandenen System.

### Start auf dem Jetson-Host

```bash
cd ~/mxck2_ws/.devcontainer
sudo docker compose -f docker-compose.all.yml up -d lidar foxglove control kickstart
```

Das passt zu deiner Compose-Datei: `lidar` startet den LiDAR-Launch, `control` ist die Shell-Basis, `kickstart` startet die Fahrkette, und `foxglove` ist separat mit Port 8765 exponiert.  

### Prüfen, dass der LiDAR-Container läuft

```bash
sudo docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}" | grep mxck2_lidar
```

### ROS-Kommandos immer mit Sourcing

Deine alten Exporte sind genau daran gescheitert. Benutze ab jetzt immer dieses Muster:

```bash
sudo docker exec -it mxck2_lidar bash -lc "source /opt/ros/humble/setup.bash && source /mxck2_ws/install/setup.bash && ros2 topic list"
```

### Kern-Tests

```bash
sudo docker exec -it mxck2_lidar bash -lc "source /opt/ros/humble/setup.bash && source /mxck2_ws/install/setup.bash && ros2 topic echo /scan --once"
sudo docker exec -it mxck2_lidar bash -lc "source /opt/ros/humble/setup.bash && source /mxck2_ws/install/setup.bash && ros2 topic hz /scan"
sudo docker exec -it mxck2_lidar bash -lc "source /opt/ros/humble/setup.bash && source /mxck2_ws/install/setup.bash && ros2 topic echo /tf_static --qos-durability transient_local --once"
```

### Erster echter Node

In `mxck_ftg_perception` baust du zuerst nur:

* `scan_front_window_check`

Aufgabe:

* `/scan` abonnieren
* optional TF nach `base_link`
* Karton vorne / links-vorne / rechts-vorne testen
* Winkel und Mindestabstand ausgeben
* optional Marker publizieren

**Akzeptanzkriterium:**

* Hindernis vorne → Winkel nahe 0
* links-vorne → positiver Winkel
* rechts-vorne → negativer Winkel

Erst wenn das stimmt, machst du weiter.

## Stage 2 — Scan-Vorverarbeitung

**Ziel:** Einen stabilen FTG-Eingang erzeugen.

Node:

* `scan_preprocessor_node`

Input:

* `/scan`

Output:

* `/autonomous/ftg/scan_filtered`
* `/autonomous/ftg/front_clearance`

Funktionen:

* NaN/Inf entfernen
* Bereich clippen
* Front-FOV ausschneiden
* optional kleine Glättung
* Scan-Timeout erkennen

**Warum diese Stage so wichtig ist:**
FTG wird instabil, wenn du direkt auf verrauschten Roh-Scans arbeitest. Das Paper arbeitet gedanklich mit sauberer geometrischer Repräsentation von Hindernisabständen und Gaps; praktisch brauchst du deshalb zuerst einen stabilen Scan-Zweig. 

**Akzeptanzkriterium:**

* `/autonomous/ftg/scan_filtered` ist stabil
* `front_clearance` reagiert logisch auf Kartonabstände
* keine NaN/Inf im FTG-Zweig

## Stage 3 — Obstacle-Layer nur optional

Hier weiche ich bewusst leicht von deiner ursprünglichen Idee ab.

### Meine klare Empfehlung:

**Für die erste fahrende FTG-Version ist ein eigener Obstacle-Node optional.**

Warum?

* In deinem Referenz-Workspace gibt es zwar bereits `obstacle_msgs`, `obstacle_substitution` und `mxck_obstacle_substitution`, also eine komplette Obstacle-Schiene.  
* Aber genau diese zusätzliche Ebene macht die erste Integration unnötig schwer.
* Für FTG brauchst du anfangs nur den **freien Raum im gefilterten Scan**.

### Was ich stattdessen empfehle

Wenn du “Obstacle” unbedingt sichtbar haben willst, dann zuerst nur als Debug:

Node:

* `obstacle_debug_node`

Output:

* `/autonomous/ftg/markers`

Also keine eigenen neuen Messages in Stage 3.

### Später optional

Erst wenn FTG schon fährt:

* `mxck_ftg_obstacles`
* und dann entweder `MarkerArray` oder Wiederverwendung des vorhandenen `obstacle_msgs` aus der Referenz

## Stage 4 — FTG-Planer

Node:

* `ftg_planner_node`

Input:

* `/autonomous/ftg/scan_filtered`

Outputs:

* `/autonomous/ftg/gap_angle`
* `/autonomous/ftg/target_speed`
* `/autonomous/ftg/markers`

Interne Rechenschritte:

1. Front-FOV lesen
2. Sicherheitsblase/Bubble um das nächste Hindernis
3. freie Segmente/Gaps bestimmen
4. größtes sinnvolles Gap wählen
5. Gap-Zentrum bestimmen
6. daraus Zielwinkel berechnen
7. zunächst einfache Speed-Policy aus Clearance ableiten

Das passt direkt zur Struktur des FGM-Papers: Gap-Array, Maximum Gap, Gap Center, Final Heading. 

### Sehr wichtige Vereinfachung für Version 1

Noch **kein** globales Ziel, noch **keine** Mischlogik mit Navigator.

Erstmal:

* “freiestes sinnvolles Gap”
* und leichte Geradeaus-Präferenz

So bekommst du eine erste robuste lokale Hindernisvermeidung.

**Akzeptanzkriterium:**

* Gerade freie Strecke → `gap_angle` nahe 0
* mittiges Hindernis → sinnvolle Ausweichrichtung
* keine wilden Sprünge im Winkel

---

## Stage 5 — Command-/Safety-Node

Node:

* `ftg_command_node`

Inputs:

* `/autonomous/ftg/gap_angle`
* `/autonomous/ftg/target_speed`
* `/autonomous/ftg/front_clearance`

Output:

* **`/autonomous/ackermann_cmd`** (`AckermannDriveStamped`) 

Aufgaben:

* steering clamp
* speed clamp
* Stop bei zu kleinem Abstand
* Stop bei Scan-Timeout
* Stop bei fehlendem Planner-Output
* optional kleines steering smoothing

### Warum separat?

So bleibt dein FTG-Planer rein algorithmisch, und der Command-Node macht nur:

* Sicherheitslogik
* Ackermann-Mapping
* finalen Systemoutput

Das ist deutlich wartbarer.

## Stage 7 — Eigener Runtime-Container für FTG

Weil eure Architektur laut README auf **one concern per container** ausgelegt ist, ist für die fertige FTG-Integration ein eigener Container sauberer als “irgendwo im control-Container mitlaufen lassen”. 

### Daher meine Empfehlung

Erweitere später `~/mxck2_ws/.devcontainer/docker-compose.all.yml` um einen neuen Service:

```yaml
auto:
  <<: *common-setup
  container_name: mxck2_auto
  command: bash
```

In der Entwicklungsphase startest du FTG dort manuell, später per Launch.

Vorteil:

* LiDAR bleibt LiDAR
* Control bleibt Control
* FTG wird ein eigenes Modul
* sauberer als alte SSH-Starter-Ideen

## Stage 9 — Realtest

Reihenfolge:

1. Nur Sensoren
2. Nur Planner-Ausgabe beobachten
3. Nur `/autonomous/ackermann_cmd` beobachten
4. dann Räder frei / Fahrzeug aufbocken
5. dann sehr langsam fahren
6. erst mittiges Hindernis
7. dann links/rechts
8. dann Engstelle

Immer:

* geringe Geschwindigkeit
* RC/Deadman verfügbar
* nie direkt an VESC vorbei
* immer durch die bestehende Control-Kette

Das ist exakt die richtige Sicherheitsphilosophie für dein Fahrzeug.
